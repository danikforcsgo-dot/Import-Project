Мой бот сам открывает сделки на бирже по телеграмм сигналам. Сам скрипт ищет на графиках точки входа по стратегии и когда
находит сигнал,то сразу отправляет сообщение в телеграм и бот сразу открывает сделку на бирже 
Вот описание стратегии:
KokojamboTrade — Торговая стратегия
Общая концепция
Автоматизированная торговая система на криптовалютном рынке, работающая на бирже BingX. Бот сканирует 103 торговые пары (токен/USDT) в режиме реального времени и автоматически открывает, усредняет и закрывает позиции по заранее заданным правилам. Результаты отображаются в реальном времени на дашборде KokojamboTrade и приходят уведомлениями в Telegram-канал @SlowbroTradingBot.

Технический анализ — условия входа
Таймфрейм: 4-часовые свечи (4H)

Индикаторы:

EMA 20 (быстрая экспоненциальная скользящая средняя)
EMA 80 (медленная экспоненциальная скользящая средняя)
ADX 14 (индекс направленного движения — сила тренда)
ATR 14 (средний истинный диапазон — волатильность)
Условия для ЛОНГА (покупка):

Цена пересекает EMA 20 снизу вверх (на предыдущей свече цена была ниже EMA 20, на последней закрытой — выше)
EMA 20 находится выше EMA 80 (подтверждение восходящего тренда)
ADX ≥ 15 (есть сила тренда, не флэт)
Условия для ШОРТА (продажа):

Цена пересекает EMA 20 сверху вниз
EMA 20 находится ниже EMA 80 (подтверждение нисходящего тренда)
ADX ≥ 15
Кулдаун: одна и та же пара не генерирует новый сигнал чаще, чем раз в 4 часа (один 4H-кандл).

Управление позицией
Плечо: ×15

Размер каждого входа: 10% от баланса счёта (залог)
→ Итоговый размер позиции = 10% баланса × 15 = 150% баланса в контракте

Стоп-лосс: цена входа ± 5.5 × ATR (рассчитывается, отображается в уведомлении — но не выставляется на бирже, мониторинг ручной)

Тейк-профит: цена входа ± 7.0 × ATR (аналогично — ориентир)

Соотношение Risk/Reward: 1 : 1.27 (5.5 ATR риск → 7.0 ATR прибыль)

DCA — усреднение позиции
Если позиция уходит в минус — бот автоматически усредняет (докупает/досдаёт) по стратегии DCA:

Максимум входов: 10 (включая первоначальный)
Интервал между входами: не чаще 1 раза в час
Размер каждого DCA-входа: 10% баланса (аналогично первому входу)
Итого: при 10 входах задействован весь баланс счёта (100%)
Условие DCA: позиция в отрицательном P&L на момент проверки
Мульти-позиция
Бот может держать до 3 позиций одновременно, при условии:

Все уже открытые позиции находятся в прибыли (unrealized P&L > 0) на BingX
Только тогда открывается новая позиция по новому сигналу
Мониторинг и частота
Частота сканирования: каждые 60 секунд (все 103 пары)
Длительность одного скана: ~60–70 секунд
Мониторинг позиций: каждые 5 секунд (DCA, баланс)
Ежечасные обновления P&L: отправляются в Telegram для открытых позиций
Ежедневный отчёт: в 23:59 МСК — итоги дня
Ретроспектива сигналов: каждые 12 часов (00:00 и 12:00 UTC) — статистика точности
Уведомления (Telegram)
При каждом событии бот отправляет сообщение в канал:

Сигнал — токен, направление, цена, EMA, ADX, ATR, SL/TP
Открытие позиции — токен, вход, залог, размер позиции, баланс, номер входа DCA
DCA-вход — добавленный объём, средняя цена, итоговый залог, текущий P&L
Закрытие позиции — итоговый P&L
Ретроспектива — сколько сигналов сработали за 12 часов

1. Структура — pnpm Monorepo
Один репозиторий, три части:

scanner.py — Python-бот (в корне)
artifacts/api-server/ — Node.js/TypeScript API (Express)
artifacts/dashboard/ — React + Vite фронтенд
Общие библиотеки в lib/ (схема БД, API-типы, клиент).

2. Зависимости

Python (pyproject.toml):

ccxt, numpy, pandas, psycopg2-binary, requests
JS (корень) — только typescript + prettier. Каждый артефакт имеет свой package.json (React, Vite, Express, Drizzle ORM и т.д.)

3. Запуск на Replit — 4 отдельных процесса (Procfile нет)

Процесс	Команда
Scanner Bot	python3 scanner.py
API Server	pnpm --filter @workspace/api-server run dev
Dashboard	pnpm --filter @workspace/dashboard run dev
Mockup Sandbox	pnpm --filter @workspace/mockup-sandbox run dev
Все управляются через Replit Workflows.

4. Объём данных в PostgreSQL — минимальный

Таблица	Строк
signals	5
scanner_status	1
live_trading_state	0 (данные в JSON-файле)
sub/paper_trading_state	0
БД практически пустая — основное хранение идёт через live_trading.json на диске.
